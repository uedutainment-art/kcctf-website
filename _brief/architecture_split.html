📌한눈 결론

결론: 분리하세요. 그것도 가장 단순한 방식 — 두 개의 별도 repo로.
  • 플랫폼 (현재 폴더): 그대로 두고 5/12 얼리버드 오픈 준비 진행. 손대지 말 것.
  • 홈페이지 (새 폴더): 별도 repo로 새로 빌드. V1 디자인, Next.js 14, Vercel.
  • 두 앱은 도메인 + URL 링크로만 연결. 코드 공유 없음.
  • 홈페이지의 "예약하기" CTA → 플랫폼의 신청 페이지 URL로 점프.
왜 monorepo 아니고 완전 분리인가? 플랫폼은 이미 production 운영 단계 (5/12 D-3), 홈페이지는 새로 빌드. 두 앱의 라이프사이클·릴리스 주기·기술 요구가 완전히 다름. monorepo로 묶는 것은 나중에 정말 필요할 때 하는 게 맞음. 지금 묶으면 오버엔지니어링.

핵심 발견 — 사용자 직관이 옳음

"플랫폼은 이벤트 운영을 다룬다. 홈페이지는 마케팅을 다룬다. 둘은 본질이 다르다."

Eventbrite / Splash / Lu.ma처럼 이벤트 운영 SaaS는 별도 도메인 + 마케팅 사이트는 별도가 업계 표준. 플랫폼이 multi-tenant 구조 (events/[slug])라면 더더욱 마케팅과 분리 — 다른 페스티벌도 같은 플랫폼 위에서 운영 가능해지므로.

②왜 지금 분리해야 하는가 — 6가지 이유

1. 라이프사이클이 완전히 다르다
홈페이지는 주마다 변경 (가격, 라인업, 카피, 사진, 회차). 플랫폼은 안정성·안전성 중심 — 한 번 잘 만들면 거의 안 건드림. 두 주기를 한 배포에 묶으면 마케팅 변경이 플랫폼 안정성을 위협하고, 플랫폼 수정이 마케팅 deploy를 막음.
2. 책임의 경계가 다르다
홈페이지 = 디자인 + 콘텐츠 + SEO + 컨버전. 디자이너·운영진의 영역.
플랫폼 = auth + DB + API + SMS + 결제 + 보안. 엔지니어의 영역.
한 코드베이스 안에 있으면 누가 무엇을 책임지는지 흐려짐.
3. 기술 스택이 다르다
홈페이지 = 정적 SSG + 빠른 빌드 + CDN + 가벼운 JS. Lighthouse 90+ 목표.
플랫폼 = SSR + DB + API + auth + 서버 로직. 보안·트랜잭션 중요.
한 앱에 두 종류 페이지를 섞으면 둘 다 절충됨 — 마케팅도 무거워지고 플랫폼도 정적화 어려움.
4. 트래픽 패턴이 다르다
홈페이지 = 피크 사용 (홍보 시작·기사 노출 시 폭주). 익명 사용자 많음.
플랫폼 = 지속적 적은 트래픽 (운영자 위주).
합쳐 있으면 마케팅 폭주가 운영 어드민 응답 지연 유발 가능.
5. 재사용성·확장성
플랫폼이 multi-tenant (events/[slug] 구조)라면 — 다른 페스티벌도 입주 가능. "엘마르", "마리포사", "DITM" 등 한국 탱고 행사들이 같은 플랫폼 안에 입주하면 SaaS가 됨. 이때 각 페스티벌 마케팅 사이트는 자기들 톤·도메인으로 별도가 정답. 플랫폼은 backend, 마케팅은 frontend per event.
6. 5/12 얼리버드 오픈이 코앞이다
플랫폼은 production 가까움 (D-3). 지금 손대면 안 됨. 그런데 새 홈페이지(V1 디자인)는 시간 들여 만들어야 함. 완전 분리하면 플랫폼은 그대로 5/12 오픈, 홈페이지는 따로 천천히 빌드. 한 repo에 합치면 — 새 홈페이지 작업이 플랫폼 안정성 흔들 위험.

공유 데이터 — 어디가 진실인가

진실의 출처 = 운영기준.md (Cowork 폴더의 마스터 문서)

가격·라인업·일정·장소 등은 운영기준.md를 보고 두 앱에 각자 하드코딩. 자주 바뀌지 않으니 (얼리버드 마감 같은 건 한 번 정해지면 끝) 수동 동기화로 충분.

나중에 필요하면: 플랫폼이 가격·일정 API 제공 → 홈페이지가 빌드 시 fetch. 또는 Sanity·Notion API 같은 headless CMS. 지금은 오버엔지니어링.

유저 흐름

1. 방문자가 kcctf.org 접속 → 마케팅 사이트 (V1 디자인) 노출
2. "예약하기" 클릭 → apply.kcctf.org/events/chuncheon-2026/register 이동
3. 플랫폼 폼 작성 (전화·국적·역할 등) → 입금 안내
4. 입금 확인 후 운영자가 어드민에서 승인 → 큐알 발급 → SMS/이메일
5. 행사장 입장 시 플랫폼 큐알 검증
6. 행사 종료 후 플랫폼은 데이터 archive, 홈페이지는 "2026 다시 보기" 페이지로 변환

⑥마이그레이션 단계 (이번 주 + 다음 주)

핵심 원칙: 플랫폼은 5/12 얼리버드 오픈까지 절대 손대지 않는다. 홈페이지는 별도로 만들고, 5/12 이후에 도메인 swap 고민.
1
이번 주 (5/9~5/11) — 플랫폼 안정화 우선
플랫폼은 발신번호 정책 결정(시나리오 A vs B) + E2E 테스트 + 얼리버드 오픈 준비. 홈페이지 작업은 새 폴더에서 병렬 진행이지만 플랫폼 코드는 절대 안 건드림.
2
5/12 얼리버드 오픈 (월) — D-Day
플랫폼 기존 도메인(kcctf.org/현 사이트)으로 그대로 오픈. 이때 마케팅은 현재 사이트 그대로.
이 시점에 새 홈페이지는 아직 라이브 X — 스테이징에만.
3
5/12 이후 — 홈페이지 본격 빌드
얼리버드 발신번호 안정화 후, 새 폴더 ~/projects/kcctf-website/에서 V1 디자인 새 사이트 작업. Claude Code로 진행. CLAUDE.md 따라가면 자동.
4
홈페이지 빌드 완료 → 스테이징에서 운영팀 비교
kcctf-2026.vercel.app로 배포 → 운영팀 검수.
이때 플랫폼은 kcctf.org/register?slug=chuncheon-2026으로 라이브 운영 중.
5
도메인 분리 — 두 단계로
5-1. 플랫폼을 apply.kcctf.org로 옮김 (서브도메인 DNS 추가).
5-2. kcctf.org를 새 홈페이지(Vercel)로 변경.
플랫폼의 기존 URL 패턴(events/[slug])은 apply.kcctf.org/events/[slug]로 자연스럽게 이전.
6
기존 폼 URL Redirect 설정
이미 발송된 카톡·홍보물의 kcctf.org/... 링크가 새 도메인으로 자동 리디렉트되도록. Vercel rewrites 또는 Next.js redirects 활용.
7
홈페이지 → 플랫폼 CTA 연결
홈페이지 "예약하기" 버튼이 apply.kcctf.org/events/chuncheon-2026/register으로 점프하도록. 서브도메인 + URL 파라미터로 자연스럽게.

⑦위험 요소 + 완화책

위험 완화책
도메인 변경 시 SEO 일시 하락 301 redirect 정확히 설정. Google Search Console에 새 사이트 제출. 한 달이면 회복.
이미 발송된 홍보물·SMS의 kcctf.org/... 링크가 끊김 Vercel rewrites로 옛 경로(/register 등)를 새 플랫폼 URL로 자동 리디렉트.
플랫폼 도메인 이전 시 다운타임 5/12 얼리버드 오픈 후, 트래픽 적은 시간(새벽)에 DNS 변경. TTL 짧게 설정.
가격·일정 정보가 두 앱에 따로 — 불일치 위험 운영기준.md를 master로. 두 앱 변경할 때 같은 PR 또는 같은 commit message에 cross-ref. 또는 하나로 옮길 때 함께.
